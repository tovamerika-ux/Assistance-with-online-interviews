# Правила конвертации DOCX → Markdown для GitHub

Этот файл описывает все проблемы, обнаруженные при конвертации Word-документов в Markdown (через pandoc), и правила оформления, применённые в данном проекте.

---

## 1. Ошибки конвертации pandoc (DOCX → MD)

### 1.1. Ложные блоки цитат (`>`)

**Проблема:** Pandoc интерпретирует отступы Word (left indent ≥ 700 twips) как Markdown-цитаты (`>`). В оригинальных Word-файлах нет стиля «Цитата» — только ручные отступы.

**Масштаб:** 663 строки в 4 файлах.

**Решение:** Убрать все `>` в начале строк. Если в документе есть настоящие цитаты со стилем Word «Quote» / «Цитата», их нужно оставить.

---

### 1.2. Экранирование спецсимволов

**Проблема:** Pandoc экранирует символы, которые имеют значение в Markdown: `\)`, `\-`, `\[`, `\]`, `\>`, `\|`. В обычном тексте это лишнее, а внутри bat-скриптов ломает код (например, `>nul` превращается в `\>nul`).

**Решение:** Удалить обратный слэш перед `)`, `-`, `[`, `]`, `>`, `|` во всём документе. Исключение — случаи, когда экранирование нужно намеренно (см. раздел 2.2).

---

### 1.3. Фраза «Откройте область навигации в Word»

**Проблема:** Pandoc вставляет артефакт из навигационной панели Word.

**Решение:** Удалить все вхождения.

---

### 1.4. Ссылки на локальные файлы и Яндекс.Диск

**Проблема:** Pandoc сохраняет ссылки из Word: `file:///D:\YandexDisk\...`, `https://disk.yandex.ru/...`, а также пути вида `\bats for calls\...`.

**Решение:** Удалить или заменить на относительные ссылки GitHub-репозитория.

---

### 1.5. Ссылки на `.docx` файлы

**Проблема:** Перекрёстные ссылки между Word-документами вида `Звонки с телефона через ПК _v3.9.docx#_Гарнитура._Динамики_молчат`.

**Решение:** Заменить на ссылки между .md файлами: `звонки-через-пк.md#гарнитура-динамики-молчат`.

---

### 1.6. HTML-комментарии `<!-- -->`

**Проблема:** Pandoc вставляет пустые HTML-комментарии.

**Решение:** Удалить все `<!-- -->`.

---

### 1.7. HTML-теги `<figure>`, `<figcaption>`, `<span class="mark">`

**Проблема:** Pandoc оборачивает изображения в `<figure>` с подписями в `<figcaption>`. GitHub отображает их, но они избыточны.

**Статус:** Оставлены как есть — GitHub рендерит их корректно.

---

### 1.8. Атрибут `style` на изображениях

**Проблема:** Pandoc генерирует `<img style="width:Xin;height:Yin" />`. GitHub **игнорирует** атрибут `style`, поэтому размеры не применяются, картинки отображаются в полном размере или с нарушением пропорций.

**Решение:** Заменить `style="width:Xin;height:Yin"` на `width="Npx"` (только ширина, без высоты — браузер сохранит пропорции автоматически). Пересчёт: `pixels = inches × 96`.

---

### 1.9. Разорванные строки в bat-коде

**Проблема:** Pandoc разбивает длинные строки bat-скриптов на несколько строк и добавляет пустые строки между каждой командой. Код становится нечитаемым и не работает при копировании.

**Решение:** Склеить разорванные строки, убрать лишние пустые строки, обернуть в кодовые блоки ` ```bat `.

---

### 1.10. Разорванные строки в обычном тексте (word wrap)

**Проблема:** Pandoc переносит текст на ~72 символа, разбивая предложения на несколько строк. В исходном Markdown это незаметно, но в некоторых контекстах (внутри списков, таблиц, inline-форматирования) GitHub может отрендерить перенос как разрыв строки.

**Масштаб:** 118+ строк в 4 файлах.

**Решение:** Склеить строки, где продолжение начинается с отступа и строчной буквы (или знака препинания). Не склеивать строки, заканчивающиеся на `.`, `!`, `)`, `:` — это настоящие концы предложений.

---

### 1.11. HTML-таблицы вместо Markdown

**Проблема:** Pandoc генерирует `<table>` с HTML-тегами вместо Markdown-таблиц.

**Статус:** Оставлены как есть — GitHub рендерит HTML-таблицы.

---

## 2. Правила оформления MD для GitHub (данный проект)

### 2.1. Нумерованные списки — экранирование

**Проблема:** GitHub Markdown (CommonMark) распознаёт `1)` и `1.` в начале строки как элемент нумерованного списка и автоматически добавляет отступ. Но подпункты вида `1.1)` не распознаются — отсюда разнобой в отступах.

**Правило:** Экранировать все `N)` и `N.` в начале строки обратным слэшем:
```
1\) Текст пункта
1\. Текст пункта
```
GitHub отобразит `1)` / `1.` без лишнего отступа.

---

### 2.2. Подпункты `N.N)` — буллет-список

**Проблема:** GitHub не поддерживает кастомную нумерацию вида `1.1)`, `1.2)`. Такие строки отображаются как обычный текст без отступа.

**Правило:** Оформлять подпункты как элементы буллет-списка с 3 пробелами отступа:
```
1\) Главный пункт
   - 1.1) Подпункт
   - 1.2) Подпункт
```

---

### 2.3. Компактные списки

**Проблема:** Если между элементами списка есть пустая строка, GitHub рендерит «свободный список» с большими межстрочными интервалами (как отдельные абзацы).

**Правило:** Между элементами одного списка **не ставить пустые строки**:
```markdown
<!-- ПРАВИЛЬНО (компактно): -->
- Элемент 1
- Элемент 2
- Элемент 3

<!-- НЕПРАВИЛЬНО (разреженно): -->
- Элемент 1

- Элемент 2

- Элемент 3
```

---

### 2.4. Якорные ссылки

**Проблема:** GitHub генерирует якоря из заголовков по правилам:
1. Перевод в нижний регистр
2. Пробелы → дефисы
3. Удаление спецсимволов (точки, скобки, кавычки и т.д.)
4. Номер раздела сохраняется

Pandoc генерирует якоря без номеров разделов и с точками, которые GitHub удаляет.

**Правило:** Якорь для заголовка `## 3. Шумодав от Krisp` = `#3-шумодав-от-krisp` (с цифрой, без точки).

**Проверка:** Открыть документ на GitHub, навести на заголовок — появится иконка ссылки, по которой можно скопировать правильный якорь.

---

### 2.5. Bat-скрипты — кодовые блоки

**Правило:** Все bat/cmd скрипты оборачивать в:
````
```bat
@echo off
REM Команды...
```
````

Внутри кодового блока:
- Символы не экранируются (Markdown их не интерпретирует)
- `>nul`, `|`, `%` отображаются корректно
- GitHub подсвечивает синтаксис

---

### 2.6. Inline-команды

**Правило:** Одиночные команды или имена переменных в тексте оборачивать в обратные кавычки:
```
Используйте `reg add "HKCU\AppEvents\Schemes"` для отключения звуков.
Символ `|` является пайпом.
```

---

### 2.7. Горизонтальные разделители

**Правило:** Между секциями настроек (WINDOWS, Krisp, Voicemeeter, Discord и т.д.) ставить `---`:
```markdown
---
###### WINDOWS

- Default device: **Voicemeeter Input**

---
###### Krisp

- **Microphone Input**: USB Headset
```

---

### 2.8. Цветовая группировка (замена цветного фона Word)

**Проблема:** GitHub Markdown не поддерживает цветной фон текста.

**Правило:** Использовать цветные эмодзи-маркеры для группировки одинаковых схем:
```markdown
- 🟢 3.1. (Phone Link + Discord; HW: no audio interface, 2x BT-W5)
- 🟣 3.1B. (Phone Link + Discord; HW: no audio interface, 1x BT-W5)
- 🔵 3.2. (Phone Link + Discord; HW: audio interface, 1Mii BT, BT-W5)
- 🟡 3.3. (Phone Link + Discord; HW: audio interface, 2x BT-W5)
```

Конфигурации одного цвета = одинаковые схемы. Конфигурации без маркера = базовые (уникальные).

---

### 2.9. Выделение текста (замена жёлтого маркера Word)

**Проблема:** GitHub Markdown не поддерживает highlight/подсветку текста.

**Альтернативы:**
- **Жирный**: `**текст**`
- **Инлайн-код**: `` `символ` `` — для выделения конкретных символов, переменных, команд
- **Скриншот из Word** — если нужно точное выделение жёлтым, вставить картинку

---

### 2.10. Изображения — только `width`

**Правило:** Указывать только `width` без `height`:
```html
<img src="../images/папка/media/image1.png" width="500" />
```
Браузер автоматически рассчитает высоту с сохранением пропорций.
