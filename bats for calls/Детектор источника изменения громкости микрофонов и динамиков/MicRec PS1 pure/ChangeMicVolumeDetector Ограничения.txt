Ограничения скрипта ChangeMicVolumeDetector.ps1

Ограничение точности определения источника:
Скрипт не может гарантировать 100% точное определение процесса, который изменил уровень громкости микрофона. Причина в ограничениях Windows Core Audio API: уведомление IAudioEndpointVolumeCallback::OnNotify сообщает факт изменения уровня и передаёт AUDIO_VOLUME_NOTIFICATION_DATA, но не содержит PID или имени процесса-источника. Поле guidEventContext позволяет лишь отличать собственные изменения приложения от внешних, если вызывающий код вообще передал свой GUID. :contentReference[oaicite:0]{index=0}

Почему текущая эвристика может ошибаться:
Текущая версия скрипта делает вывод о likely source по косвенным признакам: активные audio sessions, процессы в системе, активное окно настроек Windows, состояние Krisp и т.д. Это приближённая оценка. Дополнительно Windows документация прямо предупреждает, что перечислитель сессий IAudioSessionEnumerator / IAudioSessionManager2::GetSessionEnumerator может не знать о новых audio sessions, поэтому если опираться только на перечислитель, результаты могут быть неполными или неточными. Для более точной работы приложение должно самостоятельно поддерживать список текущих sessions через уведомления.

Что это значит на практике:
Если одновременно запущены Krisp, Teams, Voicemeeter, Яндекс Телемост, Phone Link, браузеры и окно mmsys.cpl, скрипт может показать только наиболее вероятный источник с некоторым уровнем уверенности (confidence), но не абсолютно доказанного виновника. Поэтому выводы вида "likely source" и "heuristic" нужно воспринимать как диагностическую подсказку, а не как строгое доказательство.

Рекомендации по улучшению точности:
Для версии 2.0 рекомендуется вынести детекцию в отдельный C# helper. Такой helper должен:
1. Подписываться на IAudioEndpointVolumeCallback и получать событие изменения уровня в момент его возникновения.
2. Использовать собственный guidEventContext при восстановлении уровня скриптом, чтобы надёжно отличать свои изменения от внешних.
3. Поддерживать собственный список audio sessions через IAudioSessionNotification и IAudioSessionManager2, а не полагаться только на GetSessionEnumerator.
4. Работать в MTA-потоке, так как Microsoft требует инициализацию COM в Multithreaded Apartment для корректного получения session notifications.
5. Передавать PowerShell-скрипту уже агрегированную диагностическую информацию: external/self change, likely source, confidence, active sessions, foreground process. Это не делает определение источника абсолютно точным, но заметно повышает точность по сравнению с чистым PowerShell polling.

Даже C# не даст “магических 100%”, если опираться только на IAudioEndpointVolumeCallback::OnNotify, потому что в AUDIO_VOLUME_NOTIFICATION_DATA есть guidEventContext, mute, volume и каналы, но нет PID процесса-источника. То есть C# нужен не потому, что он внезапно откроет недоступное поле, а потому что позволяет:

- ловить событие мгновенно, а не polling’ом;

- надёжно помечать свои изменения через guidEventContext;

- держать собственный актуальный список audio sessions через notifications, а не только snapshot перечислителя;

- уменьшить количество ложных эвристик.

Когда нужен ещё более глубокий разбор:
Если требуется максимально точная техническая диагностика, рекомендуется использовать ETW/WPR/WPA трассировку во время воспроизведения проблемы. Microsoft описывает Windows Performance Recorder и Windows Performance Analyzer как инструменты для записи и анализа ETW-событий, а для multimedia/audio сценариев рекомендует собирать trace именно во время ручного воспроизведения проблемы. Это подходит для сложных случаев, когда эвристика скрипта недостаточна.
